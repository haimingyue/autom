# Atoms 系统架构

## 1. 架构目标

Atoms 的系统架构需要支持以下核心目标：

- 支撑微信小程序首发
- 快速完成 MVP 开发与联调
- 保证习惯主链路稳定：登录、创建习惯、打卡、专注、进度、反思
- 为内容和监督能力预留扩展空间

因此当前采用：

- 前端：uniapp 单代码基座
- 后端：Spring Boot 单体应用
- 数据库：MySQL
- 接口形式：REST API

## 2. 总体架构

```text
微信小程序端（uniapp）
        |
        | HTTPS REST API
        v
Spring Boot 单体后端
        |
        v
      MySQL
```

说明：

- 前端统一通过 API 调用后端
- 后端统一处理鉴权、业务逻辑、数据持久化
- 数据统计初期可实时聚合，不强依赖额外缓存和异步计算

## 3. 前端架构

前端基于 `autom-front-end/` 目录中的 uniapp 工程。

### 3.1 页面层

建议按业务模块组织页面，而不是按纯技术层级组织：

- 首页 Home
- 习惯创建 / 编辑
- 习惯详情
- 进度页 Progress
- 反思页
- 专注页 Focus
- 用户页 / 个人资料页
- 登录页或登录引导页

### 3.2 前端模块层

建议至少拆分为以下业务模块：

- `auth`：登录、token、用户信息
- `habit`：习惯创建、编辑、列表、详情
- `checkin`：打卡提交与历史
- `focus`：专注记录
- `progress`：概览、连续天数、完成率
- `reflection`：反思提交与查询

### 3.3 基础能力层

- 统一请求封装
- token 注入
- 错误码处理
- 空状态和加载状态处理
- 时间和日期处理

### 3.4 状态管理建议

优先保持轻量，不提前引入复杂状态系统。

建议管理的核心状态：

- 当前登录用户
- 登录 token
- 今日习惯列表
- 进度总览摘要

## 4. 后端架构

后端采用 Spring Boot 单体架构，按业务模块分层组织。

### 4.1 分层建议

- Controller：HTTP 接口层
- Service：业务逻辑层
- Repository / Mapper：数据访问层
- Domain / Entity：领域对象与实体
- DTO / VO：请求与响应对象

### 4.2 业务模块

MVP 阶段核心模块：

- `auth`
- `user`
- `habit`
- `habit-log`
- `focus-session`
- `progress`
- `reflection`

二期预留模块：

- `content`
- `accountability`

### 4.3 单体优先原因

- 当前项目仍处于 MVP 早期
- 业务复杂度不足以支持微服务拆分
- 习惯、打卡、统计之间数据耦合很强
- 单体更适合快速开发与联调

## 5. 核心数据流

### 5.1 登录数据流

1. 小程序获取微信 `code`
2. 前端调用登录接口
3. 后端换取微信身份信息
4. 后端创建或更新用户
5. 返回 token 和用户基础资料

### 5.2 习惯创建数据流

1. 前端提交习惯表单
2. 后端校验字段
3. 落库 habit
4. 返回习惯详情

### 5.3 今日习惯查询数据流

1. 前端请求今日习惯
2. 后端基于用户时区与重复规则计算今天应展示的习惯
3. 返回今日习惯列表和基础状态

### 5.4 打卡数据流

1. 用户提交某个习惯的当日完成
2. 后端校验该日是否已完成
3. 写入 habit_log
4. 返回更新后的 streak 和完成状态

### 5.5 专注数据流

1. 用户开始专注
2. 后端创建 focus_session
3. 用户完成专注
4. 后端更新 focus_session
5. 如用户选择记录习惯，则将 focus_session 关联到 habit_log

### 5.6 进度与反思数据流

1. 前端请求进度概览
2. 后端基于 habit_log 聚合 streak、完成率、热力图
3. 用户提交反思
4. 后端写入 reflection
5. 进度页再次查询时合并展示反思结果

## 6. 数据模型

MVP 建议至少包含以下核心实体：

### user

- 用户基础信息
- 登录来源标识
- 时区
- 创建时间

### habit

- 习惯行为文本
- 身份目标文本
- 完整句式
- 重复规则
- 目标值与目标单位
- 提醒配置
- 状态

### habit_log

- 习惯 ID
- 用户 ID
- 记录日期
- 是否完成
- 完成来源
- 关联专注记录 ID

### focus_session

- 用户 ID
- 关联习惯 ID
- 计划时长
- 实际时长
- 开始时间
- 结束时间
- 完成状态

### reflection

- 用户 ID
- 习惯 ID，可为空
- 反思类型
- 问题文本
- 回答文本
- 创建时间

## 7. 关键技术决策

### 7.1 鉴权

- MVP 采用登录后 token 鉴权
- 除登录接口外，业务接口默认要求登录

### 7.2 时间与自然日

- 所有“今日”“连续天数”“打卡日期”都按用户时区计算
- 同一个习惯在同一个自然日只允许一条完成型打卡记录

### 7.3 streak 计算

- streak 以 `habit_log` 为唯一事实来源
- 前端不负责 streak 计算，只负责展示

### 7.4 统计策略

- MVP 阶段允许实时聚合
- 如果后续数据量增长，再补统计汇总表或异步任务

### 7.5 扩展策略

- 内容和监督功能在架构上预留模块
- 在没有真实业务压力前，不提前抽象为独立服务

## 8. 接口与响应约定

后端建议采用统一响应体，例如：

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

约定：

- `code = 0` 表示成功
- 非零表示业务错误或鉴权错误
- HTTP 状态码用于表示通用协议层错误

## 9. 非功能要求

- 小程序端接口响应要尽量轻量
- 页面首屏依赖的数据接口要避免过重聚合
- 错误信息要可追踪，但对前端返回要保持稳定结构
- 用户反思内容需按用户隔离，避免越权读取

## 10. 当前推荐落地方式

近期建议实际按以下顺序实现：

1. 初始化 Spring Boot 工程
2. 建立 `user`、`habit`、`habit_log`、`focus_session`、`reflection` 基础表
3. 先做认证、习惯、打卡接口
4. 再做专注、进度、反思接口
5. 最后再补内容和监督模块
