# Atoms 产品规格说明

## 1. 项目概述

**产品名称：** Atoms

**产品类型：** 一款面向移动端的习惯养成、自我提升、专注执行与复盘反思应用。

**项目判断依据：** 根据 `picture/` 中的界面图，Atoms 的核心是帮助用户建立小习惯、进行专注训练、查看行为进展，并通过内容学习与监督机制强化长期行为改变。

## 2. 问题定义

用户难以长期养成习惯，通常有以下原因：

- 目标过大，不够具体
- 日常执行不稳定
- 缺乏可见的进展反馈
- 很少进行系统性复盘
- 市面上的习惯产品通常只做打卡，不连接专注、认知、反思与监督

Atoms 希望通过一个完整闭环来解决这些问题：习惯设定、日常执行、专注支持、进度可视化、反思记录、外部监督。

## 3. 产品目标

帮助用户持续形成微小但可重复的习惯，降低每日行动阻力，并通过连续反馈让用户逐步成为自己想成为的人。

核心日循环应尽可能简单：

1. 选择或查看今天的习惯
2. 开始执行
3. 如有需要，进入专注计时
4. 完成后记录
5. 查看连续天数与进度
6. 做简短反思并继续下一天

## 4. 目标用户

- 想建立日常规律的人
- 对自我提升、行为改变、身份认同养成有兴趣的人
- 需要通过连续打卡和可视化反馈保持动力的人
- 适合通过伙伴监督提升执行率的人

## 5. 核心产品原则

- 从小开始：习惯必须足够具体、足够容易开始
- 降低摩擦：创建、打卡、查看进度都要尽量少步骤
- 强化身份认同：不仅记录行为，还要连接“我想成为什么样的人”
- 进展可见：连续天数、完成率、历史记录必须清晰
- 支持专注执行：对需要时间投入的习惯，提供专注计时能力
- 内容服务于行动：内容模块是辅助，不应喧宾夺主

## 6. 核心用户流程

### 新用户流程

1. 用户进入应用，看到欢迎或引导内容
2. 用户创建第一个习惯
3. 习惯采用结构化表达，例如：
   “我将在每天早上 8 点穿上跑鞋，这样我能成为一个健康的人。”
4. 用户设置重复日期、目标值、提醒时间
5. 创建成功后返回首页，看到今天的习惯卡片

### 日常使用流程

1. 用户打开首页，查看今天需要完成的习惯
2. 用户可直接打卡，或先进入专注计时
3. 用户完成后记录习惯完成状态
4. 用户看到里程碑反馈、连续记录或完成率更新
5. 用户进入进度页查看历史与反思
6. 用户可进入 Mindset 模块阅读成长内容

## 7. 导航结构

根据界面推断，底部导航包含四个主入口：

- 首页
- 进度
- 认知 / 心态内容
- 专注

## 8. 功能模块

### 8.1 首页

**目标：**
作为用户每日习惯执行的主入口。

**能力：**

- 展示当前日期与习惯安排
- 展示一个或多个习惯卡片
- 支持紧凑视图 / 完整视图切换
- 新建习惯
- 进入习惯详情
- 展示欢迎卡片或引导卡片

**界面特征：**

- 习惯卡片同时包含行为描述与身份目标
- 顶部有日期切换条
- 有新增习惯入口
- 有视图切换开关

### 8.2 习惯创建与设置

**目标：**
让用户用结构化方式定义一个可执行的习惯。

**必要字段：**

- 行为描述
- 身份 / 目的描述
- 重复周期
- 每日目标值
- 目标单位，例如次数或时长
- 习惯时间
- 是否提醒

**规则：**

- 用户可通过句式模板创建习惯
- 支持选择每周重复日或每日重复
- 开启提醒后，可在习惯时间发送提醒

### 8.3 习惯详情

**目标：**
查看某个习惯的完整状态与历史数据。

**能力：**

- 展示完整习惯描述
- 展示当前执行计划
- 展示累计完成次数
- 展示历史热力图 / 日历记录
- 展示完成率
- 展示当前连续天数
- 支持分享
- 进入该习惯的反思区域

### 8.4 进度页

**目标：**
集中展示用户习惯形成过程中的核心进展指标。

**能力：**

- 展示当前连续天数
- 展示历史最佳连续天数
- 展示近期完成趋势
- 展示按习惯或身份维度的反思
- 在反思不足时给出空状态提示
- 展示阶段性里程碑，例如“连续 3 天”

### 8.5 反思系统

**目标：**
让用户不只完成动作，还能形成主观感受与认知变化记录。

**能力：**

- 习惯反思
- 身份反思
- 最近 / 全部视图切换
- 无足够反思时的空状态展示

**示例问题：**

- 你如何看待自己在形成这个习惯上的进展？
- 你在成为自己想成为的人这件事上进展如何？

### 8.6 Mindset 内容库

**目标：**
通过短内容和学习材料支持用户的行为改变。

**能力：**

- 每日课程或每日内容卡片
- 内容分类浏览
- 收藏 / 保存内容
- 分类页，例如：
  - 生产力
  - 自我提升
  - 目标设定
  - 拖延
  - 意志力
  - 动机
- 展示预计阅读时长

### 8.7 专注模块

**目标：**
为需要沉浸执行的习惯提供计时支持。

**能力：**

- 默认 25 分钟专注计时
- 开始专注
- 音乐播放设置
- 结束后记录今日习惯
- 专注前选择关联习惯
- 支持通知提醒

### 8.8 监督伙伴

**目标：**
通过外部监督提升习惯坚持率。

**能力：**

- 邀请他人
- 输入邀请码
- 在进度页或个人页展示监督伙伴模块

### 8.9 个人资料

**目标：**
展示用户基础信息与总结性状态。

**能力：**

- 展示用户名
- 展示活跃习惯数量
- 展示身份数量
- 展示 Hall of Fame，用于展示已形成或已归档的习惯
- 进入监督伙伴区域

## 9. 数据模型

### 用户

- id
- display_name：显示名称
- timezone：时区
- notification_preferences：通知偏好
- created_at：创建时间

### 习惯

- id
- user_id：所属用户
- action_text：行为文本
- identity_text：身份文本
- full_statement：完整句式
- repeat_days：重复日期
- target_value：目标值
- target_unit：目标单位
- reminder_enabled：是否开启提醒
- reminder_time：提醒时间
- status：状态
- created_at：创建时间
- archived_at：归档时间

### 习惯记录

- id
- habit_id：习惯 id
- date：记录日期
- completed：是否完成
- completion_source：完成来源
- focus_session_id：关联专注记录
- created_at：创建时间

### 反思记录

- id
- user_id：用户 id
- habit_id：可为空，对应具体习惯
- reflection_type：反思类型
- prompt_text：问题文本
- answer_text：回答内容
- score：可选评分
- created_at：创建时间

### 专注记录

- id
- user_id：用户 id
- habit_id：可为空，关联习惯
- planned_duration_minutes：计划时长
- actual_duration_minutes：实际时长
- completed：是否完成
- started_at：开始时间
- ended_at：结束时间

### 内容条目

- id
- category：分类
- title：标题
- summary：摘要
- reading_time_minutes：阅读分钟数
- body：正文
- saved_by_user：是否被用户收藏

### 监督邀请

- id
- sender_user_id：发送者
- receiver_user_id：接收者，可为空
- invite_code：邀请码
- status：状态
- created_at：创建时间
- accepted_at：接受时间

## 10. 功能需求

### MVP 必须具备

- 创建、编辑、归档习惯
- 首页展示每日习惯
- 记录习惯完成状态
- 展示连续天数与完成率
- 展示历史完成记录
- 提供 25 分钟专注计时器
- 将专注记录与习惯关联
- 创建并查看习惯反思
- 提供基础内容库
- 提供个人资料与监督邀请码流程

### 建议具备

- 进度分享卡片
- 收藏文章
- 最佳连续天数记录
- 已完成习惯的 Hall of Fame
- 首页紧凑模式与完整模式切换

### 后续增强项

- 智能提醒
- 基于目标的习惯建议
- 更丰富的数据分析
- 更完整的监督互动能力

## 11. 非功能需求

- 以移动端体验为优先
- 每日核心动作必须足够快
- 尽可能支持弱网或离线记录
- 提醒送达要稳定
- 反思内容需注意隐私保护
- 新用户首次使用门槛要低

## 12. 成功指标

- 习惯创建转化率
- 第 1 天、第 7 天、第 30 天留存
- 活跃用户平均完成习惯数
- 每周反思填写率
- 活跃用户平均专注次数
- 连续记录维持率
- 监督邀请接受率

## 13. 风险点

- 连续天数机制如果过于强调，可能对中断用户造成挫败感
- 内容模块如果权重过高，可能分散用户对核心执行流程的注意力
- 监督功能需要足够简单，否则使用率会低
- 反思如果出现时机不对，用户可能不会填写

## 14. 待确认问题

- 产品首屏引导是以“习惯”为中心，还是以“身份认同”为中心？
- 内容模块是人工编辑、AI 生成，还是混合模式？
- 专注模式是通用计时器，还是必须绑定习惯？
- 监督伙伴是否包含实时聊天，还是只做邀请和状态同步？
- Hall of Fame 只是归档区，还是包含奖励机制？

## 15. 推荐 MVP 范围

第一版建议仅交付以下能力：

- 账号与个人资料
- 习惯创建与每日首页
- 完成记录
- 连续天数与历史记录
- 专注计时器
- 习惯反思
- 静态内容库
- 监督邀请码流程，不做聊天

## 16. 总结

Atoms 是一款围绕“微习惯形成”设计的移动产品。它的核心差异点在于把以下能力放进同一个闭环里：

- 基于身份认同的习惯设定
- 清晰的连续记录与历史追踪
- 内置专注计时
- 反思与复盘机制
- 自我提升内容支持
- 轻量级监督伙伴机制

如果按当前截图继续推进实现，Atoms 应优先保证“创建习惯、执行习惯、记录习惯、看到进展”这条主链路足够顺畅。
